1. OVERVIEW
================================================================================

Portfolio X-Ray is a deterministic mechanical service for Brazilian investment
portfolios. It does NOT contain AI logic. It:

  1. Maintains 5 reference tables describing the Brazilian investment universe
  2. Accepts a portfolio CSV (positions mapped to instrument templates)
  3. Resolves each position against the reference tables
  4. Produces structured precursors for downstream AI report generators

The service has 242 instrument templates covering: government bonds, CDBs,
LCIs, LCAs, CRAs, CRIs, debentures, listed equities, BDRs, ETFs, FIIs,
investment funds, crypto, and COEs.

economic_exposure_weights_json must be present. Weights must sum to 1.0.

3. ENDPOINTS — INSTRUCTION TEXT (Batch Write)
================================================================================

Each of the 5 tables has 6 instruction text fields, one per report angle:
  instructions_diversification
  instructions_liquidity
  instructions_fees
  instructions_tax
  instructions_risk
  instructions_performance

These are free-text fields that downstream AI agents will receive as context
when generating report sections. They can be updated individually via PATCH
on any record, or in bulk:

PATCH /api/v1/batch/instructions/
{
  "angle": "diversification",
  "updates": [
    {"table": "economic_exposure", "code": "fi_cdi_br", "text": "CDI-indexed instruments..."},
    {"table": "tax_profile", "code": "general", "text": "General tax regime..."},
    {"table": "instrument_template", "code": "cdb_cdi_generic", "text": "CDB specifics..."}
  ]
}

Valid angles: diversification, liquidity, fees, tax, risk, performance
Valid tables: economic_exposure, tax_profile, issuer_type, issuer, instrument_template

The operation is atomic: if any entry fails validation, no changes are applied.
Set "text" to null to clear a field.

6. CSV INPUT CONTRACT
================================================================================

Required columns:
  instrument_template_code   — must exist in instrument_template table
  weight                     — positive decimal, all rows must sum to 1.0 (±0.0005)
  expiry_date                — YYYY-MM-DD, required if template.requires_expiry_date=true
  issuer_code                — optional, must exist in issuer table if provided
  issuer_type_code           — optional fallback if issuer_code absent

Example:
  instrument_template_code,weight,expiry_date,issuer_code,issuer_type_code
  cdb_cdi_generic,0.30,2027-06-15,banco_btg_pactual_s_a,
  tesouro_selic_generic,0.20,2029-01-01,,
  equity_vale3,0.15,,,
  fii_hglg11,0.10,,,
  fund_btg_cdb_plus,0.25,,,

================================================================================
7. CONVENTIONS
================================================================================

- All codes are lowercase snake_case, immutable after creation
- No DELETE endpoints — use PATCH {"active": false} to deactivate
- Duplicate POST returns 409 with existing record (idempotent)
- Foreign keys: submitted as code strings on write, returned as {code, name} on read
- All list endpoints paginated: ?page=1&page_size=50
- Names appear alongside codes in all outputs (display_name preferred)
- The processing pipeline is deterministic: same CSV + same DB = identical output

================================================================================
8. ERROR RESPONSES
================================================================================

All errors follow:
{
  "error": "validation_error|conflict|not_found|method_not_allowed",
  "message": "Human-readable description",
  "field_errors": {"field_name": ["error detail"]},      // for validation
  "existing": {...},                                       // for 409 conflict
  "references": [...]                                      // for deactivation blocked
}

401: Missing or invalid token
400: Validation error (bad input)
404: Resource not found
409: Conflict (duplicate POST or deactivation blocked by references)
