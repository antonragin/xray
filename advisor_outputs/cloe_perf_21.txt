Yes — and the key design choice is this:

**Do not make a “performance” precursor. Make a `performance_context` precursor.**

Because with zero returns, zero benchmarks, and zero price history, the system cannot talk about realized performance.  
What it *can* talk about is:

1. **what economically drives returns**,  
2. **what rate/inflation/equity/FX regimes matter**,  
3. **where maturity/mark-to-market sensitivity sits**,  
4. **which market proxies are the right contextual lens in Brazil**.

---

## Short answers to your 5 questions

### 1. How should precursor be structured?
Use **4 layers**:

1. **Data availability / limitations**
   - explicitly say there is no realized performance data
   - no benchmark series
   - no market prices
   - only composition, exposures, maturity, instructions

2. **Aggregated return-driver exposures**
   - aggregate instrument rows into portfolio-level economic drivers
   - e.g. `fi_cdi_br`, `fi_ipca_br`, `eq_br`, `eq_international`, etc.
   - this becomes the core of the section

3. **Maturity / sensitivity context**
   - summarize known tenor buckets
   - identify where mark-to-market sensitivity is likely higher
   - distinguish short CDI carry vs long IPCA+/prefixado duration risk

4. **Narrative guardrails**
   - what the writer/model may say
   - what it must not say
   - how proxies should be framed

---

### 2. Should it include “return driver descriptions”?
**Yes — absolutely.**  
That is the most useful thing this precursor can provide.

Without returns, the performance section should basically answer:

> “If this portfolio performs well or poorly, what will most likely explain it?”

So each driver should have:
- a **human-readable description**
- a **Brazilian contextual proxy/reference**
- a **plain-language behavior statement**
- optional **sensitivity flags** like inflation, real rates, nominal rates, FX, risk appetite, credit spread

---

### 3. Should it explicitly state “DO NOT include specific return numbers”?
**Yes. Strongly yes.**

And not just in prose — make it **machine-readable**.

You want explicit rules like:
- do not state realized return
- do not state benchmark return
- do not state excess return / alpha
- do not imply backfilled performance
- do not say “outperformed” or “underperformed”
- do not attach numeric return assumptions to proxies

Important nuance:

- **Forbidden:** return numbers, benchmark numbers, alpha, Sharpe, drawdown
- **Allowed:** weights, maturity buckets, years to expiry, exposure percentages

So the rule is really:

> **Do not include specific performance numbers. Exposure and maturity numbers are allowed.**

---

### 4. Brazilian context: CDI, IPCA+, Ibovespa, S&P 500 in BRL?
**Mostly yes, with one important refinement: treat them as contextual proxies, not formal benchmarks unless you actually know the benchmark.**

Recommended mapping:

- `fi_cdi_br` → **CDI / Selic**
- `fi_ipca_br` → **IPCA + real rate curve**  
  contextual lens: Tesouro IPCA+, NTN-B, IMA-B-style family
- `fi_prefixado_br` → **nominal pre-fixed curve**
- `eq_br` → **Ibovespa**  
  pragmatic default in Brazil
- `eq_international` → **S&P 500 in BRL terms** *if US-heavy exposure*  
  otherwise use “global equities + FX” framing
- `re_br` → **IFIX**
- `crypto` → **Bitcoin in BRL**
- `fi_international` → **global bonds + FX (if unhedged/hedge unknown)**

### Important nuance on CDI
In Brazil, **CDI is the universal opportunity-cost reference**, but it should **not** be treated as the natural benchmark for every asset class.

So:
- good as the main lens for `fi_cdi_br`
- okay as a “cash baseline” for the whole portfolio
- **not ideal** as the primary driver proxy for equities, FIIs, crypto, or international assets

# Strong recommendation: use “proxy” or “context_reference”, not “benchmark”

This matters a lot.

If you do not actually know the benchmark, **don’t call it benchmark**.  
Call it one of:

- `context_reference`
- `market_proxy`
- `return_driver_proxy`
- `reference_lens`

Why?

Because “benchmark” implies:
- a formal mandate benchmark
- or a valid performance comparison series

You do **not** have that.

So the correct language is:

- **CDI is the reference lens for the CDI-linked sleeve**
- **Ibovespa is the contextual market proxy for Brazilian equities**
- **S&P 500 in BRL is a contextual proxy for US-heavy offshore equity exposure**

Not:
- “the benchmark is CDI”
- “the portfolio underperformed Ibovespa”
- “returns should match S&P 500 in BRL”
