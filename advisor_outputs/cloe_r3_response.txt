## Must-fix before sign-off

### 1) `simulate_clients.py` likely has a real chart bug with NumPy arrays
In `make_trajectory_chart()`:

```python
x=time_years + time_years[::-1]
y=pct[hi] + pct[lo][::-1]
```

If `time_years` / `pct[...]` are NumPy arrays, `+` does **elementwise addition**, not concatenation. That will silently produce the wrong fan-band polygon.

Also these checks are unsafe for NumPy arrays:

```python
if stats.get("benchmark_median_path"):
if not counts or not mids:
```

They can raise `ValueError: truth value of an array is ambiguous`.

**Fix:** explicitly convert to lists and use length/None checks.

Example pattern:
```python
time_vals = list(time_years)
hi_vals = list(pct[hi])
lo_vals = list(pct[lo])

x_fill = time_vals + time_vals[::-1]
y_fill = hi_vals + lo_vals[::-1]

bm = stats.get("benchmark_median_path")
if bm is not None and len(bm) > 0:
    ...
```

**Severity:** blocker, because it can silently make the charts wrong.

### 2) Benchmark series may be misaligned with each portfolio series
You build benchmark once:

```python
cdi_portfolio = engine.build_portfolio_returns(data, {"CDI": 1.0})
```

Then for each account you simulate using:

```python
portfolio["portfolio_returns"]
portfolio["ipca"]
benchmark_returns=cdi_portfolio["portfolio_returns"]
```

If `portfolio["months"]` is shorter/different from the full CDI series, the benchmark may be misaligned or length-mismatched.

**Fix:** build or slice the CDI benchmark to the **same months** as the client portfolio.

**Severity:** blocker for simulation correctness.

### 3) Silent dropping + renormalization is still financially dangerous
You fixed renormalization after dropping unavailable indices, which is good mechanically.  
But this is still not enough for production.

Today the logic does this:

- drop indices not in `data["indices"]`
- renormalize remaining weights to 1.0
- simulate using the **full account value**

That means if 20% of the portfolio is unmapped/unavailable, you are effectively pretending the remaining 80% explains 100% of the money.

**Minimum safe behavior:**
- compute `coverage = covered_brl / total_brl`
- record dropped indices and values
- either:
  - **skip simulation** if coverage is below a threshold (e.g. 95% / 98%), or
  - simulate only covered amount and disclose that clearly

Right now this can materially distort outputs while looking clean.

**Severity:** blocker.

---

### 4) Direct fixed-income mapping still has a meaningful gap: inflation-linked debentures
In `Renda Fixa`:

```python
if "INFLACAO" in est:
    if prod_type in ("CRA", "CRI", "CDCA"):
        return ("IDA-IPCA", ...)
    return ("IMA-B", ...)
```

So an inflation-linked `DEB` falls to `IMA-B`, which is a **government real-rate proxy**, not a corporate inflation credit proxy.

At minimum, `DEB` should probably join the corporate IPCA bucket:

```python
("CRA", "CRI", "CDCA", "DEB")
```

And infrastructure/incentivada debentures should have a path to `IDA-Infra`.

**Severity:** high.  
This is exactly the kind of misclassification that can overstate duration beta and understate credit spread risk.

### 5) You still rely too much on exact string matches / fixed schema
A few examples:

- `mercado == "Conta Corrente"`
- `prod_type == "NTNB"`
- `ativo == "IVVB11"`

This is brittle.

**Recommended minimum:**

- keep raw values only for display
- validate the header row instead of assuming fixed column positions
- fail fast if required columns are missing/moved

**Severity:** high for production robustness.

## Financial mapping accuracy: what still looks wrong or fragile

### A) `IDA-DI` is still a catch-all bucket
Your overall summary is:

- **IDA-DI = 58.3%**
- **Ibovespa = 0.1%**
- **ACWI ≈ 0**

That may be real, but it’s also the bucket absorbing many approximations:
- crédito privado funds
- pós corporate bonds
- real-estate credit funds
- some debenture-like assets
- possibly DEBB11

So I would **manually inspect the top 10–20 largest IDA-DI lines** before sign-off.  
That’s the bucket most likely to hide residual mapping error.

### B) Direct infra / incentivada bond mapping is underdeveloped
You handle infra well for funds, but for direct `Renda Fixa` instruments the logic is much thinner. If the source file contains:
- debênture incentivada
- debênture infraestrutura
- IPCA+ infra bonds

you likely won’t map them as well as you should.

**Suggestion:** use `submercado`, `produto`, `ativo`, and maybe `emissor` text together for RF mapping, not just `estrategia`.

### C) Gold / crypto / uranium → `SP500TR` is still only acceptable as an explicit proxy
This is fine only if everyone agrees it is a **risk proxy**, not an economic proxy.

- Gold does **not** behave like S&P 500
- Crypto does **not** behave like S&P 500
- Uranium/miners do **not** behave like S&P 500

I wouldn’t block launch over this if:
- exposures are small
- `[APPROX]` is visible
- simulation coverage is disclosed
- stakeholders accept proxy logic

But if these sleeves are material, this remains too rough.

### D) Cash → CDI is a business assumption, not a fact
For brokerage cash / transit values, `CDI` is a reasonable convenience proxy.  
But if these balances are truly idle and non-remunerated, you are overstating expected return.

Not a coding bug, but you should document the assumption.

You do:

### 7) `POSITION_DATE` is hard-coded

### 8) Split mappings should be validated
If a future rule accidentally returns weights summing to 0.9 or 1.1, the code will accept it silently.

Add a guard:
```python
if isinstance(mapping, dict):
    s = sum(mapping.values())
    if abs(s - 1.0) > 1e-6:
        raise ValueError(...)
```

Probably fine for numeric accounts, but production code should sanitize and deduplicate.
