================================================================================
1. PURPOSE
================================================================================

Build 6 independent report services — one per analysis angle — that take a
Portfolio X-Ray precursor as input and produce a polished HTML client report
as output. Each service makes exactly ONE call to an AI advisor (Cloe or Diana)
to assemble the narrative portion of the report.

The services are developed iteratively: 10 rounds per angle, 2 advisor calls
per round (one to review, one to implement improvements), across all 6 angles
= 120 advisor calls for development.

Report generation calls (the one advisor call inside each service) are separate
and only run on a golden set during development.

================================================================================
2. INVESTMENT PHILOSOPHY (HARD CONSTRAINT)
================================================================================

All reports MUST conform to the passive wealth management philosophy:

REQUIRED:
  - Analyze what the client CURRENTLY holds and its structural characteristics
  - Frame observations in portfolio theory terms (diversification, systematic
    risk, fee drag, tax efficiency)
  - Present factual data: allocations, coverage, maturity profiles, fee totals
  - Identify structural concerns (concentration, liquidity mismatches, tax
    inefficiency) without recommending specific trades
  - Use hedging language: "the portfolio exhibits", "this allocation implies"
  - When data is missing or partial, say so explicitly

STRICTLY PROHIBITED:
  - Recommending specific securities to buy or sell
  - Suggesting the client can "beat the market" or achieve alpha
  - Stock picking, market timing, or tactical allocation suggestions
  - Speculative statements about future returns, prices, or market direction
  - Inventing specific numbers not present in the precursor data
  - Claiming one asset class will outperform another
  - Any language implying active management superiority

LOCALE:
  - Reports are for Brazilian clients. Use Brazilian conventions:
    * Currency: R$ with period thousands separator, comma decimal (R$ 1.234,56)
    * Dates: DD/MM/YYYY
    * Tax references: IR, IOF, come-cotas, PGBL/VGBL
    * Regulatory: CVM, ANBIMA, B3, FGC
    * Benchmarks: CDI, IPCA, Ibovespa, Selic
  - Narrative text should be in Portuguese (pt-BR)
  - Table headers and labels may be bilingual (pt-BR primary)

COMPLIANCE FOOTER (required on every report):
  "Este relatório é meramente informativo e não constitui recomendação de
  investimento. Rentabilidade passada não é garantia de resultados futuros.
  Consulte seu assessor de investimentos antes de tomar decisões."

================================================================================

3.2 REPORT SERVICE INTERFACE

Each service.py MUST implement:

  def generate_report(precursor: dict, angle: str, client_name: str) -> str:
      """
      Args:
          precursor: full JSON from POST /api/v1/process/
          angle: one of diversification/liquidity/fees/tax/risk/performance
          client_name: for display

      Returns:
          Self-contained HTML string
      """

The function MUST:
  1. Deterministically compute angle-specific metrics in Python (NOT the LLM)
  2. Assemble a bounded prompt with relevant facts + instructions
  3. Make EXACTLY ONE advisor call
  4. Require the advisor to return STRUCTURED JSON (not HTML, not freeform)
  5. Validate the JSON response
  6. Render final HTML deterministically from the template + JSON narrative

The advisor NEVER writes HTML. The advisor returns JSON like:
  {
    "executive_summary": "O portfólio apresenta...",
    "sections": [
      {"title": "Alocação por Classe", "body": "A carteira está concentrada..."},
      {"title": "Concentração por Emissor", "body": "O maior emissor..."}
    ],
    "key_observations": ["observation 1", "observation 2"],
    "caveats": ["Data de emissores disponível para apenas 71% da carteira"]
  }

FALLBACK: If the advisor call fails, times out, or returns invalid JSON,
the service MUST produce a deterministic fallback report using only the
computed metrics and data tables (no narrative sections).

3.3 DATA FLOW

  Client CSV
      ↓
  X-Ray API (POST /api/v1/process/)
      ↓
  Precursor JSON (bundle + angles + instructions)
      ↓
  metrics.py (deterministic computation: coverage, concentration, weighted
              averages, top-N lists, bucketing, FGC analysis, etc.)
      ↓
  Prompt assembly (metrics + relevant instructions + house rules)
      ↓
  ONE advisor call → structured JSON narrative
      ↓
  HTML template rendering (Jinja2, deterministic)
      ↓
  Self-contained HTML report

KEY PRINCIPLE: The LLM narrates pre-computed facts. It does NOT compute
arithmetic, generate tables, write HTML, or create charts. All quantitative
work is done in Python before the call.

4. X-RAY SERVICE API REFERENCE
================================================================================

Base URL: https://xray-oryx.onrender.com
Auth: Authorization: Bearer OryxRulezzz2026!!!

4.1 PROCESS PORTFOLIO
  POST /api/v1/process/
  Body: {"csv": "instrument_template_code,weight,...\n..."}
  Returns: {run_id, validation, bundle, angles}

  The response contains:
    bundle.portfolio_summary   — row_count, coverage metrics
    bundle.allocations         — by exposure, issuer_type, issuer, kind, maturity
                                  (each entry has code, name, weight)
    bundle.positions           — resolved data, derived fields, instructions
    angles[angle]              — position-level data + instructions for each angle

4.2 READ DB STATE
  GET /api/v1/economic-exposures/          (13 rows, single page)
  GET /api/v1/tax-profiles/                (4 rows)
  GET /api/v1/issuer-types/                (6 rows)
  GET /api/v1/issuers/?page_size=100       (212 rows — MUST paginate: follow 'next')
  GET /api/v1/instrument-templates/?page_size=100  (242 rows — MUST paginate)

  IMPORTANT: page_size max is 200. For tables with >200 rows, follow the
  'next' URL until it is null.

4.3 UPDATE INSTRUCTIONS (the ONLY X-Ray mutation allowed)
  PATCH /api/v1/batch/instructions/
  Body: {"angle": "<angle>", "updates": [{"table": "...", "code": "...", "text": "..."}]}

  IMPORTANT: After ANY instruction update, affected client precursors become
  stale. You MUST re-run POST /api/v1/process/ to get fresh precursors
  before generating reports.

8. ITERATION PROTOCOL (PER ANGLE)
================================================================================

For each of the 6 angles, in order:
  diversification → liquidity → fees → tax → risk → performance

ROUND 1 (Bootstrap):

  Step 1: Build initial service
    - Create service.py, metrics.py, config.json, report.html template
    - metrics.py computes ALL quantitative outputs deterministically:
      * Allocation percentages (from precursor)
      * Coverage metrics
      * Concentration metrics (top-1, top-3, top-5 per dimension)
      * Weighted averages where applicable
      * FGC coverage computation
      * Maturity profile statistics
    - service.py assembles prompt from: metrics + instructions + house rules
    - Advisor call returns structured JSON with narrative sections
    - Template renders HTML from metrics + narrative JSON

  Step 2: Generate reports for golden set
    - Run on 4-6 golden clients only
    - Save HTML reports

  Step 3: Review (ADVISOR CALL 1 of 2)
    Submit to the review advisor (opposite of the service advisor):
    - 2-3 sample precursors (golden set)
    - Generated HTML reports for those clients
    - Complete service.py + metrics.py + template code
    - Current instructions_* text for this angle (from core taxonomy)
    - Full X-Ray DB state summary

    Ask for evaluation of:
    1. Report quality and insight depth
    2. Accuracy of analysis vs precursor data
    3. Instruction text effectiveness
    4. Anti-hallucination compliance
    5. Brazilian market conventions
    6. Missing insights a CPA-20 advisor would expect
    7. Specific improvements (with exact text for instruction updates)

  Step 4: Implement (ADVISOR CALL 2 of 2)
    Submit to the implementation advisor:
    - Review suggestions from Step 3
    - Current code + templates + instructions
    - Ask for: updated code, updated instructions, explanation of changes

  Step 5: Apply and regenerate
    - Update service code
    - Update instructions via PATCH /api/v1/batch/instructions/
    - Re-run POST /api/v1/process/ for golden set (fresh precursors)
    - Regenerate golden set reports
    - Save iteration artifacts

ROUNDS 2-5 (Structural refinement):
  Same Steps 2-5. Focus on:
    - Missing analysis sections
    - Bad framing or inaccurate statements
    - Instruction text quality
    - Metric computation gaps

ROUND 6 (Cross-angle regression):
  Run ALL 6 services on golden set. Review for:
    - Visual consistency across angles
    - Tonal consistency
    - Terminology consistency
    - Contradictions between angle reports
    - Repeated content across angles

ROUNDS 7-10 (Polish):
  Focus on:
    - Narrative quality and depth
    - Edge case handling (sparse portfolios, unusual allocations)
    - Anti-hallucination hardening
    - Professional tone and pt-BR language quality

9. REPORT CONTENT GUIDELINES (PER ANGLE)
================================================================================

All required items are required ONLY when supported by precursor data or
instruction text. If support is absent or partial, the report MUST state
the limitation explicitly and avoid inferring the missing fact.

9.1 DIVERSIFICATION
  Deterministic metrics (computed in Python):
    - Allocation by exposure group (%, names)
    - Allocation by economic exposure (%, names)
    - Allocation by issuer type (%, names)
    - Top 5 issuers by weight
    - Issuer coverage percentage
    - Country split (Brazil vs international, derived from exposure codes)
    - Instrument kind distribution
    - HHI or effective-N if applicable

  Advisor narrative topics:
    - Interpretation of concentration levels
    - Issuer family common-shock risks
    - FGC conglomerate exposure
    - Tax-driven concentration patterns
    - Coverage gaps and what they mean

9.2 LIQUIDITY
  Deterministic metrics:
    - Maturity bucket distribution (for dated instruments)
    - % of portfolio in listed/exchange-traded instruments
    - % with FGC coverage
    - Weighted average maturity (for fixed income)
    - % of portfolio with vs without defined maturity
    - Coverage: maturity coverage on total vs on applicable

  Advisor narrative topics:
    - Liquidity profile interpretation
    - Lockup / carência implications (from instructions)
    - Mark-to-market risk for early exits
    - Fund redemption characteristics (from instructions)

9.3 FEES
  Deterministic metrics:
    - Weighted average fee (where fee data available)
    - Fee coverage (% of portfolio with fee data)
    - Fee distribution by instrument type
    - Top 5 highest-fee positions
    - Total estimated annual fee drag (in bps)

  Advisor narrative topics:
    - Fee level interpretation vs market norms
    - Come-cotas as implicit fee drag
    - Fee-free instruments and hidden costs (spread)
    - Fee impact on long-term compounding

9.4 TAX
  Deterministic metrics:
    - Allocation by tax profile (%, names)
    - For fixed income: IR bracket distribution based on days_to_expiry
    - % subject to come-cotas
    - % tax-exempt
    - % in previdência

  Advisor narrative topics:
    - Tax regime implications
    - Come-cotas drag analysis
    - IR regressive table benefits of long-term holding
    - Tax-exempt concentration patterns
    - IOF risk for short-term positions (if purchase date available)

9.5 RISK
  Deterministic metrics:
    - Exposure group concentration
    - Single-issuer max weight
    - Credit risk tiers (government/bank/corporate/unrated)
    - Currency exposure (BR vs international %)
    - FGC coverage ratio
    - Maturity risk indicators

  Advisor narrative topics:
    - Interest rate sensitivity
    - Credit risk interpretation
    - Currency risk assessment
    - Concentration risk evaluation
    - What risk factors the portfolio is structurally exposed to

9.6 PERFORMANCE
  Deterministic metrics:
    - Exposure-based return driver identification
    - Fee drag quantification
    - Tax drag estimation (come-cotas, IR brackets)
    - Maturity profile (for fixed income return context)

  Advisor narrative topics:
    - Return driver descriptions (CDI, IPCA+spread, equity market)
    - Fee and tax drag on net returns
    - Structural return characteristics
    - EXPLICITLY: what the portfolio is NOT positioned for
    - PROHIBITED: specific return projections or numbers

10. HTML REPORT FORMAT
================================================================================

Self-contained HTML with inline CSS. No external dependencies.

Structure:
  1. Header: client name, date, angle title, firm branding
  2. Executive summary card (3-5 sentences from advisor JSON)
  3. Key metrics cards (computed deterministically)
  4. HTML/CSS bar charts (generated in Python, NOT by LLM)
  5. Data tables (allocations with names + codes + percentages)
  6. Narrative sections (from advisor JSON, rendered in template)
  7. Key observations list (from advisor JSON)
  8. Caveats and data limitations (from advisor JSON + coverage metrics)
  9. Compliance disclaimer footer (standard text, always present)

Charts: Use pure HTML/CSS horizontal bars. Example:
  <div style="background:#4A90D9; width:33%; height:24px; display:inline-block"></div>
  <span>Renda Fixa 33%</span>
No SVG, no JavaScript, no external charting libraries.

Design: Professional financial report aesthetic. Think private bank quarterly
review, not tech dashboard.

11. INSTRUCTION TEXT MANAGEMENT
================================================================================

The instructions_* fields are the ONLY part of X-Ray the agent may modify.

HIERARCHY (more specific overrides broader):
  instrument_template > issuer > issuer_type > economic_exposure > tax_profile

BOOTSTRAP (Phase 0):
  Populate all 23 core taxonomy records (13 exposures + 4 tax + 6 issuer types)
  across all 6 angles before iterative rounds begin.

FORMAT:
  [Scope] What this record represents
  [Use for] What the advisor should discuss about positions with this record
  [When material] Conditions under which this guidance is important
  [Do not infer] What the advisor must NOT assume or state
  [Brazil note] Local market/tax/regulatory nuance

RULES:
  - Keep instructions angle-specific
  - Keep instructions concise and factual
  - Do not include dynamic market views or forecasts
  - Do not duplicate guidance already in the global prompt
  - Do not encode values that exist as structured fields
  - If no record-specific instruction exists, the service must still function
    using higher-level instructions and deterministic facts
  - After any instruction update, re-run precursor generation before reports

15. HARD RULES
================================================================================

  1. Each report service makes EXACTLY ONE advisor call. Not zero, not two.
  2. The X-Ray processing engine code MUST NOT be modified. Only the
     instructions_* DB fields may be changed via the API.
  3. The advisor returns STRUCTURED JSON. HTML rendering is deterministic.
  4. All quantitative metrics are computed in Python, not by the LLM.
  5. No investment recommendations. No "buy X", "sell Y", "rebalance to Z".
  6. No invented numbers. All quantities come from precursor data.
  7. The agent MUST use the X-Ray API (not import code directly).
  8. Iterate on the golden set (4-6 clients). Full set only for regression.
  9. After instruction updates, always re-run precursor generation.
  10. Each iteration must be saved for audit.
  11. Reports must be self-contained HTML (no external dependencies).
  12. If data is missing or partial, state the limitation — don't guess.
  13. Reports in pt-BR with Brazilian financial conventions.
  14. All calls are stateless — include full context every time.
